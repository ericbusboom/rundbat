# Plan: rundbat generate / build / up / down / logs — Docker Artifact Generation and Lifecycle

## Context

The previous plan designed the configuration model (deployment targets, services, deploy modes in `rundbat.yaml`). This plan covers converting that configuration into Docker artifacts and providing lifecycle commands. The user should go from `rundbat.yaml` to a working deployment with a single `rundbat generate` followed by `rundbat up <deployment>`.

## New CLI Commands

| Command | Purpose |
|---------|---------|
| `rundbat generate` | Read `rundbat.yaml`, produce all Docker artifacts in `docker/` |
| `rundbat build <deployment>` | Build images for a deployment |
| `rundbat up <deployment>` | Deploy (or redeploy) — pull/build + start containers |
| `rundbat down <deployment>` | Stop a deployment |
| `rundbat logs <deployment>` | Tail logs from a deployment |

These replace `init-docker` as the primary generation command. `init-docker` could remain as an alias or be deprecated.

Beyond these, users can use `docker compose` directly — rundbat just generates the files and provides convenience wrappers.

## What `rundbat generate` Produces

### docker/ directory layout

```
docker/
  Dockerfile                           # App image (one per project)
  entrypoint.sh                        # Entry point — decrypt config, run server
  docker-compose.dev.yml               # Compose for dev deployment
  docker-compose.local.yml             # Compose for local deployment
  docker-compose.test.yml              # Compose for test deployment  
  docker-compose.prod.yml              # Compose for prod deployment
  .dev.env                             # Environment for dev (from dotconfig)
  .local.env                           # Environment for local
  .test.env                            # Environment for test
  .prod.env                            # Environment for prod
  .dockerignore                        # (project root, not docker/)
  Justfile                             # Recipes for all deployments
```

**No generic `docker-compose.yml`** — each deployment gets its own compose file named `docker-compose.<deployment>.yml`. This avoids ambiguity and lets each deployment declare exactly its services, ports, labels, and env file.

### Per-deployment compose files

Each `docker-compose.<name>.yml` is generated from the deployment's config in `rundbat.yaml`:

```yaml
# docker-compose.prod.yml (generated by rundbat generate)
services:
  app:
    image: ghcr.io/owner/repo:latest    # or build: context
    env_file: [docker/.prod.env]
    ports: ["${MYAPP_PORT:-8080}:8080"]
    restart: unless-stopped
    labels:
      caddy: myapp.example.com
      caddy.reverse_proxy: "{{upstreams 8080}}"

# docker-compose.local.yml (generated)
services:
  app:
    build:
      context: ..
      dockerfile: docker/Dockerfile
    env_file: [docker/.local.env]
    ports: ["${MYAPP_PORT:-8080}:8080"]
    restart: unless-stopped

  postgres:
    image: postgres:16
    env_file: [docker/.local.env]
    volumes: [postgres_data:/var/lib/postgresql/data]
    healthcheck: ...

  redis:
    image: redis:7
    healthcheck: ...

volumes:
  postgres_data:
```

Key per-deployment differences:
- **services**: only what that deployment declares (e.g., prod might be app-only, local has app+postgres+redis)
- **image vs build**: `github-actions` deployments use `image:`, others use `build:`
- **env_file**: points to `docker/.<deployment>.env`
- **labels**: Caddy labels only when `hostname` + `reverse_proxy: caddy`
- **ports**: derived from framework detection (8080 for Astro/nginx, 3000 for Node, 8000 for Python)

### Per-deployment env files

`rundbat generate` fetches environment for each deployment from dotconfig:

```python
for name, dep in deployments.items():
    if dep.get("env_source") == "dotconfig":
        env_content = config.load_env(name)  # dotconfig load -d <name> --stdout
        (docker_dir / f".{name}.env").write_text(env_content)
    else:
        # Copy from .env.example as template
        (docker_dir / f".{name}.env").write_text(generate_env_example(...))
```

These `.env` files are in `docker/` and gitignored (they contain secrets). The compose files reference them via `env_file`.

### Dockerfile

One Dockerfile per project (already exists — `generate_dockerfile()`). No change here except ensuring the entrypoint is wired in:

```dockerfile
COPY docker/entrypoint.sh /entrypoint.sh
RUN chmod +x /entrypoint.sh
ENTRYPOINT ["/entrypoint.sh"]
CMD ["npm", "start"]  # or gunicorn, nginx, etc.
```

### Entry point script

Generated based on framework. Handles:
1. Decrypt secrets if SOPS key is available
2. Run migrations if applicable
3. Exec into the server command

```bash
#!/bin/sh
set -e

# Decrypt secrets if age key is mounted
if [ -f /run/secrets/age-key ]; then
    export SOPS_AGE_KEY_FILE=/run/secrets/age-key
fi

# Framework-specific pre-start
# Django: python manage.py migrate
# Prisma: npx prisma migrate deploy
# etc.

exec "$@"
```

### Justfile

Generated with per-deployment recipes. Each deployment gets `<name>_build`, `<name>_up`, `<name>_down`, `<name>_logs`:

```just
# myapp — Docker deployment recipes

# --- dev ---
dev_up:
    docker compose -f docker/docker-compose.dev.yml up -d

dev_down:
    docker compose -f docker/docker-compose.dev.yml down

dev_logs *ARGS:
    docker compose -f docker/docker-compose.dev.yml logs {{ARGS}}

# --- local ---
local_build:
    docker compose -f docker/docker-compose.local.yml build

local_up: local_build
    docker compose -f docker/docker-compose.local.yml up -d

local_down:
    docker compose -f docker/docker-compose.local.yml down

local_logs *ARGS:
    docker compose -f docker/docker-compose.local.yml logs {{ARGS}}

# --- prod ---
prod_build:
    DOCKER_CONTEXT=docker1.example.com docker compose -f docker/docker-compose.prod.yml build

prod_up:
    DOCKER_CONTEXT=docker1.example.com docker compose -f docker/docker-compose.prod.yml up -d

prod_down:
    DOCKER_CONTEXT=docker1.example.com docker compose -f docker/docker-compose.prod.yml down

prod_logs *ARGS:
    DOCKER_CONTEXT=docker1.example.com docker compose -f docker/docker-compose.prod.yml logs {{ARGS}}

# --- database ops (if postgres) ---
local_psql:
    docker compose -f docker/docker-compose.local.yml exec postgres psql -U ${DB_USER:-app} ${DB_NAME:-app}

local_db_dump:
    docker compose -f docker/docker-compose.local.yml exec postgres pg_dump ...
```

Remote deployments prefix commands with `DOCKER_CONTEXT=<context>`. Local deployments use the default context. The `github-actions` strategy deployments use `docker compose pull` instead of `build`.

### docker run mode

When `deploy_mode: run`, instead of a compose file, `rundbat generate` stores the `docker_run_cmd` in `rundbat.yaml` (already designed in the config plan). The Justfile recipe calls it directly:

```just
prod_up:
    DOCKER_CONTEXT=docker1.example.com docker pull ghcr.io/owner/repo:latest
    DOCKER_CONTEXT=docker1.example.com docker stop myapp || true
    DOCKER_CONTEXT=docker1.example.com docker rm myapp || true
    DOCKER_CONTEXT=docker1.example.com docker run -d --name myapp \
      --env-file docker/.prod.env \
      -p 8080:8080 \
      --restart unless-stopped \
      -l caddy=myapp.example.com \
      -l "caddy.reverse_proxy={{upstreams 8080}}" \
      ghcr.io/owner/repo:latest
```

## CLI Command Details

### `rundbat generate`

1. Load `rundbat.yaml`
2. Detect framework (if not already in config)
3. For each deployment in `deployments`:
   a. Generate `docker/docker-compose.<name>.yml` with that deployment's services, build/image, env_file, labels
   b. Fetch env from dotconfig (if `env_source: dotconfig`) → write `docker/.<name>.env`
   c. If `deploy_mode: run`, generate and store `docker_run_cmd` in config
4. Generate `docker/Dockerfile` (once, framework-aware)
5. Generate `docker/entrypoint.sh` (once, framework-aware)
6. Generate `docker/Justfile` with per-deployment recipes
7. Generate `.dockerignore` in project root (if not present)
8. Print summary of generated files

Flags:
- `--json` — machine output
- `--deployment <name>` — regenerate only one deployment (not all)

### `rundbat build <deployment>`

Thin wrapper:
```python
# compose mode
docker compose -f docker/docker-compose.<deployment>.yml build

# run mode (github-actions)
# no local build — image comes from registry
```

Uses `DOCKER_CONTEXT` from the deployment config.

### `rundbat up <deployment>`

```python
# compose mode
docker compose -f docker/docker-compose.<deployment>.yml up -d

# run mode
# execute docker_run_cmd from config (after pull + stop + rm)
```

For remote deployments, prefixes with `DOCKER_CONTEXT`. For `github-actions` strategy, does `pull` before `up`.

### `rundbat down <deployment>`

```python
# compose mode
docker compose -f docker/docker-compose.<deployment>.yml down

# run mode
docker stop <app_name> && docker rm <app_name>
```

### `rundbat logs <deployment>`

```python
# compose mode
docker compose -f docker/docker-compose.<deployment>.yml logs -f

# run mode
docker logs -f <app_name>
```

## .gitignore updates

The per-deployment env files contain secrets fetched from dotconfig and **must not** be committed. The current project `.gitignore` only has `.env` (exact match at root), which won't catch `docker/.dev.env`, `docker/.prod.env`, etc.

`rundbat generate` must ensure the project `.gitignore` includes:

```
# Per-deployment env files (contain secrets from dotconfig)
docker/.*.env
```

It should append this if not already present (same pattern as how `.dockerignore` is created — check before writing).

The generated compose files, Dockerfile, entrypoint, and Justfile **should** be committed — they're the deployment spec. Only the `.env` files are secret.
